# user-rule

## 概述

在项目根目录维护 `user-rule.md`，记录用户所有要求。所有实现必须严格遵循用户的项目要求与架构原则 **违反用户要求的方案都是错误的。** 选择最符合用户要求的方案，而非最简单的方案。

## 核心流程

每次交互时自动执行（无需询问用户）：
1. 检查项目根目录是否存在 `user-rule.md`
2. 如果不存在，直接创建 `user-rule.md`
3. 将用户本次提出的所有新要求追记到 `user-rule.md`
4. 实现前先读取 `user-rule.md`，确保不违反已有要求

## user-rule.md 强制模板

创建或更新 `user-rule.md` **必须**使用下方模板。模板用 `<!---BEGIN TEMPLATE--->` 和 `<!---END TEMPLATE--->` 包裹，必须将包裹标记之间的内容写入 `user-rule.md`（按需填充占位内容）。

## 架构原则（强制遵守）
- **工程一致性**：ESLint/Prettier/TypeScript strict 自动化，禁止人工审查
- **架构即文档**：领域目录自解释，重大决策记录轻量 ADR
- **按领域组织**：`features/{domain}/` 一级目录，禁止技术类型作顶层
- **目录结构明确**：路径即边界，嵌套 ≤ 4 层
- **适度拆分**：30 秒可理解全貌，强内聚不强行拆散
- **专文件专功能**：一文件一核心职责，逻辑层独立文件
- **就近原则**：子组件/样式/测试/类型必须同目录
- **内聚优先于拆分**：被 ≥2 领域消费才提取独立
- **显式依赖**：禁止全局隐式状态，import 表达一切依赖
- **契约优先**：领域间通过 `index.ts` 类型与 API 通信
- **索引文件**：外部只引用 `index.ts`，内部直接引用真实路径
- **禁止跨领域引用**：ESLint 边界规则自动化拦截
- **构建边界**：映射 Nx/Turborepo，每个领域独立构建测试
- **扩展性**：插件化/策略模式注册，新增不改现有
- **可测试性内建**：与 UI 解耦，测试同目录就近放置
- **渐进加载**：代码分割强制，微前端非默认

## 架构原则

这些原则写入 `user-rule.md` 且不可违背：

| 原则 | 含义 | 做法 |
|------|------|------|
| **工程一致性** | 全项目强制执行统一的工程规范与代码风格，消除个人风格差异 | 必须通过 ESLint、Prettier、TypeScript strict、commitlint 配置自动化拦截；禁止依赖人工审查或自觉遵守 |
| **架构即文档** | 代码结构自解释，关键架构决策有持久化记录 | 领域目录结构必须自解释；所有领域边界定义与重大架构取舍必须记录轻量 ADR |
| **按领域组织** | 目录以业务领域为一级划分单元 | 采用 `features/{domain}/` 目录，每个领域独立包含组件、API、工具、类型；跨领域通用代码允许置于 `shared/` 内核目录，随边界清晰逐步下沉 |
| **目录结构明确** | 路径即业务边界，禁止技术类型作为一级目录 | 一级目录必须为业务领域；禁止以 `components/` / `utils/` / `hooks/` 作为顶层目录；目录嵌套深度不得超过 4 层 |
| **适度拆分** | 文件大小与认知负荷匹配，禁止过度拆分与过度膨胀 | 单文件必须在 30 秒内可理解全貌；强内聚逻辑禁止强行拆散；UI 层允许组件与专属类型/常量共存，业务逻辑层必须纵向独立拆分 |
| **专文件专功能** | 单一文件承载单一核心职责 | 一个文件仅允许一个核心职责；展示层允许合理内聚（组件 + Props 类型 + 样式常量），数据层/逻辑层必须独立文件 |
| **就近原则** | 关联代码物理位置必须相邻 | 子组件、样式、类型定义、测试文件必须与父级文件置于同一目录；禁止跨领域全局目录打散 |
| **内聚优先于拆分** | 强关联逻辑物理共存，弱耦合逻辑物理隔离 | 组件及其类型、常量、单元测试必须同目录共存；仅当逻辑被两个及以上领域消费时，才允许提取为独立文件 |
| **显式依赖** | 所有依赖关系必须通过显式声明表达 | 禁止全局隐式状态、隐式注入、魔法字符串/数字；模块间依赖必须通过 `import` 语句与文件路径表达 |
| **契约优先** | 跨模块通信必须先定义显式接口契约 | 领域间调用必须通过目标领域 `index.ts` 暴露的显式类型与 API；禁止直接引用对方内部实现文件或内部类型定义 |
| **索引文件** | 目录通过 `index.ts` 封装内部实现并控制对外接口 | 外部模块仅允许引用目录的 `index.ts`；目录内部文件之间禁止通过 barrel 文件引用，必须直接引用真实文件路径 |
| **禁止跨领域引用** | 领域之间禁止穿透内部实现 | `features/A` 仅允许引用 `features/B/index.ts`；同领域内部允许跨子目录直接引用；必须通过 ESLint 边界规则自动化拦截违规引用 |
| **构建边界** | 领域模块支持独立构建、测试与增量编译 | 目录结构必须映射构建工具（Nx / Turborepo / pnpm workspace）的模块边界；每个领域必须可独立执行构建与测试命令 |
| **扩展性** | 新增功能禁止修改现有核心代码 | 功能扩展必须通过新增文件实现；核心逻辑必须通过插件化、策略模式或配置化注册，保持对修改关闭 |
| **可测试性内建** | 可测试性作为架构设计的一等约束 | 复杂业务逻辑必须与 UI 渲染层解耦；纯函数必须覆盖单元测试，领域逻辑覆盖集成测试，关键用户流程覆盖 E2E；测试文件必须与源码同目录就近放置 |
| **渐进加载** | 按业务优先级分批加载代码 | 代码分割为强制要求；路由与大型组件懒加载为按需要求；微前端仅在组织/技术债务达到阈值时采用，非默认方案 |

## Before vs After

### Before（无 user-rule）

```
对话 1:  创建 features/auth/ → 完美
对话 10: 直接在 src/ 下平铺 5 个组件 → 已经忘了
对话 20: app.ts 3000 行，所有逻辑混一起 → 完全退化
新会话:  从零开始再犯一遍 → 无记忆
```

代码结局：

```js
// app.ts — 3000 行，每次 AI 都"先这样写，后续再改"
// 但后续永远不会来
```

### After（有 user-rule）

```
每次交互 → 读 user-rule.md → 强制对齐 → 实现
```

即使新会话，AI 依然读到完整约束：

```
对话 1:   创建 features/auth/ → 记录到 user-rule.md
对话 50:  仍读 user-rule.md → 仍用 features/auth/ → 一致
新会话:   读 user-rule.md → 从上次终点继续 → 零遗忘
```

代码结局：

```
project/
  user-rule.md                        # 持久化所有要求
  src/
    features/
      auth/
        index.ts        # 30行 — 导出接口
        login.ts        # 60行 — 登录
        register.ts     # 80行 — 注册
      upload/
        index.ts        # 20行
        handlers.ts     # 100行
      filter/
        index.ts        # 15行
        presets.ts      # 120行
```

## 优先级规则

当多个需求冲突或选择方案时：

```
用户明确要求 > 用户隐含意图 > 架构原则 > 代码简洁性 > 代码量少
```

**任何时候：** 如果某个方案让后续扩展更困难，它就是错误的方案。

## 理由对抗表

| 常见借口 | 现实 |
|---------|------|
| "项目太小不需要拆分" | 小项目也会长大。一开始就做对。 |
| "先快速实现再重构" | 重构永远不会发生。第一次就做对。 |
| "简单方案也满足当前需求" | 它不满足"扩展性强"的要求。 |
| "这次情况特殊" | 没有例外。所有项目都适用。 |
| "用户没指定结构" | 这就是为什么要有 user-rule.md — 记录隐含要求。 |
| "放 components/ 也能用" | 技术分层导致碎片化，领域组织按业务聚合更合理。 |
| "直接 import 内部文件更方便" | 破坏目录边界，后续重构成本剧增。 |
| "一个文件放所有相关代码算了" | 内聚不是堆砌。职责清晰 + 合理行数才可维护。 |

## 红色标志 — 立即停止

- 单文件堆砌多个不相关功能或混合多个职责
- 平铺所有文件在根目录或用 `components/`/`utils/` 技术分层
- 选择"简单但不满足用户要求"的方案
- 没有创建/更新 `user-rule.md`
- "这次特殊"、"先凑合"、"后续再改"
- 跨领域直接 import 内部文件（绕过 index.ts）
- 全局打散子组件/样式/测试到远处目录
- 强行拆分强内聚代码（盲目追求"小"）

**以上任意一条违规 = 删除代码，重新按照 user-rule 实现。**

## 常见错误

| 错误 | 修复 |
|------|------|
| 忘记创建 user-rule.md | 每次交互第一步检查 |
| 选择最简单方案而非最符合要求方案 | 用优先级规则决策 |
| 把所有代码放一个文件 | 按职责拆分到独立文件，但强内聚逻辑不强行拆散 |
| 用 `components/`/`utils/` 技术分层 | 改为 `features/` 领域分层 |
| 跨领域直接 import 内部文件 | 只能通过 `index.ts` 引用 |
| 子组件/测试放到远处目录 | 就近原则，放在同目录下 |
| 后续功能破坏已有结构 | 开闭原则：扩展而非修改 |
